- **Problem**: Manual `cron run` commands timeout, but scheduled crons work fine

- **Source**: Gmail hooks processing emails in parallel
- **Impact**: Session store lock contention - processes timeout acquiring `/Users/matthew/.openclaw/agents/main/sessions/sessions.json.lock`

### Why Scheduled Crons Work
- ‚úÖ Scheduled crons run independently via time-based triggers
- ‚ùå Manual `cron run` uses WebSocket RPC which is overloaded by hook processes

### Identified Solutions
1. **Immediate (bandaid)**: Kill stale hook processes with `pkill -f "openclaw.*hook"`
2. **Short-term**: Rate limit Gmail hooks - process sequentially instead of parallel
3. **Medium-term**: Increase session store lock timeout
4. **Long-term**: 
   - Debounce Gmail webhook - batch emails instead of one-process-per-email
   - Use different session store with better concurrency (Redis?)

## Gmail Hook Parallelism Fixed (10:34 AM) ‚úÖ

### Problem
- Gmail push hooks causing 38 concurrent openclaw processes
- Each incoming email triggered immediate parallel processing
- Session store lock contention at `/Users/matthew/.openclaw/agents/main/sessions/sessions.json.lock`
- Manual cron triggers timing out due to WebSocket overload

### Investigation
- Examined OpenClaw config schema for concurrency controls
- No built-in `maxConcurrent`, `debounce`, or batching options for Gmail hooks
- Push-based architecture (Google Pub/Sub) inherently creates parallel invocations

### Solution Implemented
1. **Disabled Gmail push hooks** - Removed `presets: ["gmail"]` from config

   - **Cron ID**: `3956a4f1-f07b-4ce6-869d-5d69664debb2`

3. **Restarted gateway** - Config applied cleanly via SIGUSR1

### Trade-offs
- ‚úÖ Complete concurrency control (single process)
